## 🎯 Project Mission

Build an **intelligent energy management system** for home automation that optimizes:

- Battery storage (charge/discharge timing)
- PV generation (forecasting & utilization)
- Load management (demand shifting)
- Electric vehicles & heat pumps
- Electricity pricing (minimize grid consumption, maximize feed-in revenue)

**Key Philosophy**: Simulation + Genetic Algorithm Optimization = Smart Energy Decisions

---

## 🏛️ Architecture Principles

### Modular Design

- **Separation of Concerns**: adapter, config, optimization, prediction, devices are independent
- **Pydantic for Type Safety**: All configs use Pydantic v2 with strict validation
- **Plugin Architecture**: New devices/adapters can be added without core changes

### Data Flow

```
External Source (HA, API)
    ↓
adapter/ (normalize data)
    ↓
config/ (validate & transform)
    ↓
optimization/ (genetic algorithm)
    ↓
server/ (API & Dashboard)
    ↓
External System (HA, NodeRED)
```

### Optimization Strategy

- **DEAP Genetic Algorithm**: Multi-objective optimization (cost, grid load, etc.)
- **Simulation First**: Always simulate before recommending actions
- **Forecast-Driven**: PV, weather, price forecasts shape optimization

## 🚀 Development Workflow

### Before You Code

1. Check [PROJECT_OVERVIEW.md](PROJECT_OVERVIEW.md) for architecture
2. Understand the module you're changing (read its docstrings)
3. Keep the virtual env in sync: `make update-env`

### While Coding

- **Type Hints**: Required! Helps both mypy and future developers
- **Docstrings**: RST format (used by Sphinx autodoc)
- **Constants**: Define in `config/` (not hardcoded!)
- **Tests**: Write tests as you code, run `make test`

### Before Committing

```bash
make format          # Code formatting
make mypy            # Type checking
make test            # Run all tests
make gitlint         # Check commit message
```

### Committing

- **Conventional Commits**: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:`
- Use `cz commit` for interactive commit message builder
- Commitizen auto-generates changelog

## 📊 Configuration System

### How Config Works

1. **Pydantic Models** define structure in `config/`
2. **Config Files** (YAML/JSON) are validated against models
3. **Migrations** [test_configmigrate.py](tests/test_configmigrate.py) handle version upgrades
4. **Type Safety** ensures runtime correctness

### Adding New Config Options

1. Add Pydantic field to appropriate config model
2. Provide sensible default value
3. Add validation if needed
4. Update docs in DOCS.md
5. Write test in test_config\*.py

### Config Validation

- Use Pydantic validators for complex rules
- Example: Battery min_soc < max_soc
- Run `make test` to catch validation errors early

## 🔐 Performance & Optimization

### Hot Paths (Performance Critical)

- **Genetic Algorithm Loop**: [optimization/genetic/genetic.py](src/akkudoktoreos/optimization/genetic/genetic.py)
  - Uses local vars for 3-5x speedup (see line 169 comment)
  - Minimize function calls in inner loop
- **Data Loading**: [data/](src/akkudoktoreos/data/)
  - LMDB for fast access
  - Caching with cachebox
  - Compression for large arrays

### Environment Variables (Thread Performance)

```bash
# Set in Dockerfile or docker-compose
OPENBLAS_NUM_THREADS=1
OMP_NUM_THREADS=1
MKL_NUM_THREADS=1
```

- **Critical for Home Assistant**: HA runs on single threads
- Multi-threading in optimization causes performance issues

### Profiling

- Use `make test-profile` to identify bottlenecks
- Line 169 in genetic.py shows local var optimization example

## 🤝 Contributing Guidelines

### Code Style

- **Type Hints**: Always (mypy runs in CI)
- **Formatting**: `make format` (ruff + isort)
- **Docstrings**: RST format
- **Constants**: UPPERCASE, in config files
- **Naming**: snake_case for functions/vars, PascalCase for classes

### PR Process

1. Create feature branch: `git checkout -b feat/my-feature`
2. Code with tests
3. Run full checks: `make format && make mypy && make test`
4. Commit with conventional commits
5. Push and create PR
6. Ensure CI passes
7. Wait for review

### Code Review Focus

- ✅ Tests pass locally
- ✅ Type hints complete
- ✅ Docstrings present
- ✅ No hardcoded values
- ✅ Follows architecture patterns
- ✅ Backward compatibility (if possible)

## 🆘 Quick Troubleshooting

### Common Issues

| Problem                                | Solution                                            |
| -------------------------------------- | --------------------------------------------------- |
| Tests fail locally                     | Run `make update-env` then `make test`              |
| Import errors                          | Check `.venv` is activated; run `make install`      |
| Slow optimization                      | Check env vars set; see Performance section         |
| Dashboard not updating                 | Try hardrefresh (Ctrl+F5); check console for errors |
| Config migration fails                 | Check config version in EOS.config.json             |
| Type check fails locally but not in CI | Run `make mypy` explicitly                          |
